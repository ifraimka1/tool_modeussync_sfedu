# ADR-0001: Ручное создание оцениваемых элементов Modeus через `mod_modeussync`

- Статус: принято
- Дата: 2026-07-16
- Компоненты: `tool_modeussync`, `mod_modeussync`
- Решение заменяет: автоматическое создание `mod_assign` внутри `tool_modeussync\task\pull_courses`

## 1. Контекст

`tool_modeussync` синхронизирует Moodle с Modeus и связанными интеграционными сервисами. В текущей реализации фоновая задача `pull_courses` не только создаёт курсы, но и автоматически создаёт задания, возвращённые SyncService.

Требуется изменить этот процесс: данные заданий должны сохраняться в Moodle, но создание оцениваемых элементов курса должен подтверждать преподаватель через отдельный activity-плагин `mod_modeussync`.

Новый плагин должен:

- зависеть от `tool_modeussync`;
- использовать его настройки подключения к SyncService, включая `syncservice_base_url` и `internal_api_key`;
- использовать существующий HTTP-клиент `tool_modeussync\service\SyncService` либо совместимый публичный сервис из `tool_modeussync`;
- поддерживать ручной выбор между `mod_assign` и `mod_quiz`;
- создавать пустой `quiz`, вопросы в который преподаватель добавляет самостоятельно;
- сохранять действующие контракты SyncService без изменений;
- существовать на курсе только в одном экземпляре;
- не допускать ручного добавления дополнительных экземпляров;
- идемпотентно обрабатывать повторное получение тех же данных.

## 2. Фактический текущий алгоритм

Текущий поток подтверждён кодом `pull_courses.php`, `LmsAdapterService.php` и `SyncService.php`.

### 2.1. Сессия синхронизации

Базовая задача `base_sync_job`:

1. определяет тип сессии `PULL_COURSES`;
2. получает последнюю закрытую сессию;
3. создаёт новую сессию через LmsAdapter;
4. вызывает `pull_courses::do_work()`;
5. закрывает сессию после успешного выполнения.

### 2.2. Получение и создание курсов

`pull_courses` получает прототипы курсов через:

```text
GET api/v1/lms/{lmsId}/sync-sessions/{sessionId}/sync/courses
```

Для каждого прототипа:

- существующий курс ищется по `course.idnumber = coursePrototype.id`;
- новый курс создаётся через Moodle API;
- секции и модули из исходного прототипа РМУП создаются сразу;
- Modeus ID курса извлекается из `coursePrototype.summary`;
- формируется связка Moodle и Modeus:

```json
{
  "id_lms": 123,
  "id_modeus": "modeus-course-id"
}
```

### 2.3. Получение заданий

Созданные или найденные курсы отправляются пакетами до 25 элементов:

```text
POST {syncservice_base_url}/new-course
```

Вызов выполняется методом:

```php
\tool_modeussync\service\SyncService::send_created_courses()
```

Данные заданий приходят непосредственно в ответе этого запроса, а не через отдельный endpoint. Используемая кодом структура ответа:

```json
{
  "results": [
    {
      "success": true,
      "id_lms": 123,
      "id_modeus": "modeus-course-id",
      "courseData": [
        {
          "id": "modeus-assignment-id",
          "name": "Название задания",
          "grade": 100
        }
      ]
    }
  ]
}
```

Массив `results[*].courseData` является источником заданий для курса.

### 2.4. Текущее создание заданий

Для каждого элемента `courseData` текущий код:

1. проверяет наличие `mod_assign` с таким же `course_modules.idnumber`;
2. создаёт или переиспользует секцию «Задания из Modeus»;
3. создаёт `mod_assign`;
4. записывает:
   - `courseData.id` в `course_modules.idnumber`;
   - `courseData.name` в название задания;
   - `courseData.grade` в максимальную оценку;
5. после обработки курса отправляет в SyncService связку курса через `POST /sync`.

### 2.5. Роль `ukd_grade_helper`

`tool_modeussync\classes\ukd_grade_helper` не участвует в рабочем потоке:

- класс нигде не вызывается;
- максимальная оценка берётся напрямую из `courseData.grade`;
- методы `get_max_grade()` и `get_fallback_grade()` не используются;
- настройки и file area, на которые рассчитывает helper, не подключены к текущему `settings.php`.

В рамках настоящего решения `ukd_grade_helper` не используется и не является источником данных заданий или максимальной оценки.

## 3. Проблема текущей реализации

Автоматическое создание `mod_assign` в cron-задаче не позволяет преподавателю:

- выбрать подходящий тип оцениваемого элемента;
- решить, должно ли задание быть обычным заданием или тестом;
- подтвердить создание элементов перед изменением структуры курса.

Дополнительно текущая реализация:

- не сохраняет `courseData` после завершения запроса;
- не имеет отдельных состояний для полученных и созданных заданий;
- не поддерживает частичные повторные попытки;
- жёстко привязана к `mod_assign`;
- вызывает `/sync` в рамках той же cron-задачи сразу после автоматического создания.

## 4. Принятое решение

Полученные от SyncService задания сохраняются в таблицах `tool_modeussync`. Новый `mod_modeussync` предоставляет единственный системный интерфейс курса для просмотра очереди, выбора типа элемента и ручного запуска создания.

Направление основной зависимости:

```text
mod_modeussync -> tool_modeussync -> SyncService
```

`tool_modeussync` остаётся владельцем:

- интеграционного контракта;
- данных, полученных от SyncService;
- очереди и её состояний;
- настроек подключения;
- публичного клиента SyncService.

`mod_modeussync` отвечает за:

- интерфейс преподавателя;
- создание `assign` и `quiz`;
- хранение единственного activity-экземпляра;
- запуск финального уведомления `/sync` через публичный сервис `tool_modeussync`.

## 5. Рассмотренные альтернативы

### 5.1. Хранить очередь в `mod_modeussync`

Отклонено.

`pull_courses` принадлежит `tool_modeussync`. Если он напрямую записывает данные во внутренние таблицы `mod_modeussync`, административный плагин начинает зависеть от activity-плагина, хотя activity-плагин уже зависит от административного. Это создаёт двустороннюю связанность и затрудняет обновление компонентов.

### 5.2. Хранить весь `courseData` одним JSON-полем экземпляра

Отклонено.

Один JSON не обеспечивает удобное и надёжное хранение:

- состояния каждого задания;
- выбранного типа;
- созданного `cmid`;
- частичных ошибок;
- повторных попыток;
- идемпотентного обновления отдельных заданий.

Исходный JSON разрешено сохранять дополнительно для диагностики, но не использовать как единственную модель состояния.

### 5.3. Продолжать создавать `assign` автоматически, а плагин использовать для преобразования

Отклонено.

Преобразование одного типа Moodle activity в другой небезопасно и не соответствует требованию о подтверждении до создания.

## 6. Целевой алгоритм

### 6.1. Фоновая синхронизация

`pull_courses`:

1. открывает сессию `PULL_COURSES`;
2. получает прототипы курсов из LmsAdapter;
3. создаёт отсутствующие курсы, секции и исходные модули РМУП;
4. отправляет связки созданных курсов в `POST /new-course`;
5. получает `results[*].courseData`;
6. валидирует и идемпотентно сохраняет задания в локальную очередь;
7. публикует событие о наличии заданий для курса;
8. не создаёт `assign` или `quiz`;
9. не вызывает `POST /sync`;
10. закрывает сессию синхронизации после успешного сохранения ответа.

### 6.2. Создание единственного `mod_modeussync`

`mod_modeussync` подписывается на публичное событие `tool_modeussync` о сохранении заданий. Обработчик:

1. проверяет, что для курса есть хотя бы один элемент очереди;
2. получает блокировку по `courseid`;
3. ищет существующий экземпляр `mod_modeussync`;
4. создаёт секцию «Задания из Modeus», если она отсутствует;
5. создаёт в ней один экземпляр `mod_modeussync`, если экземпляра ещё нет;
6. ничего не создаёт при повторном событии, если экземпляр уже существует.

Событие сохраняет направление зависимости: `tool_modeussync` публикует факт, не вызывая внутренние классы activity-плагина; `mod_modeussync`, который зависит от `tool_modeussync`, реагирует на событие.

Если обработчик не смог создать activity, сохранённая очередь не теряется. Следующий повторный ответ, повторное событие или административная задача восстановления снова выполняет идемпотентную проверку.

### 6.3. Действия преподавателя

Преподаватель:

1. открывает `mod_modeussync` на странице курса;
2. видит все полученные задания и их состояния;
3. для каждого ещё не созданного задания выбирает:
   - «Задание» (`assign`);
   - «Тест» (`quiz`);
4. нажимает «Создать»;
5. получает результат создания каждого элемента;
6. при частичной ошибке исправляет причину и повторяет создание;
7. после создания всех элементов видит подтверждение успешной отправки `/sync`.

## 7. Модель данных

### 7.1. Очередь курса

Таблица `tool_modeussync_course_queue`:

| Поле | Тип/ограничение | Назначение |
|---|---|---|
| `id` | primary key | Идентификатор записи |
| `courseid` | foreign key to `course.id`, unique | Moodle-курс |
| `idmodeus` | char, not null | Modeus ID курса |
| `status` | char, not null | Состояние очереди курса |
| `lasterror` | text, nullable | Последняя безопасная диагностическая ошибка |
| `timecreated` | integer | Время создания |
| `timemodified` | integer | Время последнего изменения |
| `timesynced` | integer, nullable | Время успешного `/sync` |

Уникальный индекс по `courseid` гарантирует одну очередь на курс.

Состояния:

- `pending` — есть задания, требующие действия преподавателя;
- `processing` — выполняется создание элементов;
- `awaiting_sync` — все элементы созданы, но `/sync` ещё не подтверждён;
- `synced` — `/sync` успешно завершён;
- `sync_failed` — элементы созданы, но `/sync` завершился ошибкой.

### 7.2. Элементы очереди

Таблица `tool_modeussync_queue_items`:

| Поле | Тип/ограничение | Назначение |
|---|---|---|
| `id` | primary key | Идентификатор записи |
| `queueid` | foreign key | Очередь курса |
| `externalid` | char, not null | `courseData.id` |
| `name` | char/text, not null | `courseData.name` |
| `maxgrade` | number, not null | `courseData.grade` |
| `targetmodule` | char, not null, default `assign` | Выбранный тип Moodle activity |
| `status` | char, not null | Состояние задания |
| `coursemoduleid` | integer, nullable | Созданный `course_modules.id` |
| `createdby` | foreign key to `user.id`, nullable | Пользователь, запустивший создание |
| `lasterror` | text, nullable | Последняя безопасная ошибка |
| `payloadjson` | text, nullable | Исходный элемент `courseData` для аудита |
| `timecreated` | integer | Время создания |
| `timemodified` | integer | Время последнего изменения |

Уникальный индекс:

```text
(queueid, externalid)
```

Разрешённые значения `targetmodule`:

- `assign`;
- `quiz`.

Состояния:

- `pending`;
- `processing`;
- `created`;
- `failed`.

### 7.3. Экземпляр activity

Основная таблица `modeussync`:

| Поле | Тип/ограничение | Назначение |
|---|---|---|
| `id` | primary key | Идентификатор экземпляра |
| `course` | foreign key to `course.id`, unique | Moodle-курс |
| `name` | char, not null | Название activity |
| `intro` | text | Описание |
| `introformat` | integer | Формат описания |
| `timecreated` | integer | Время создания |
| `timemodified` | integer | Время изменения |

Уникальный индекс по `course` является обязательной защитой от второго экземпляра, включая конкурентные запросы.

## 8. Правила идемпотентного обновления

Идентичность задания определяется парой:

```text
Moodle course.id + courseData.id
```

При повторном получении `courseData`:

- новый `externalid` создаёт новый элемент со статусом `pending` и типом `assign`;
- для `pending` или `failed` обновляются `name`, `maxgrade` и `payloadjson`;
- ранее выбранный преподавателем `targetmodule` сохраняется;
- для `created` обновляется диагностический `payloadjson`, но Moodle activity автоматически не изменяется;
- существующая запись не дублируется;
- отсутствие ранее полученного задания в новом ответе не удаляет запись и созданный activity;
- появление нового задания после состояния `synced` переводит очередь обратно в `pending`;
- полностью идентичный повторный ответ не изменяет `synced` и не создаёт новый экземпляр плагина.

Перед созданием выполняется дополнительная проверка Moodle:

```text
course_modules.course = courseid
course_modules.idnumber = externalid
```

Если элемент уже существует:

- `assign` или `quiz` связывается с записью очереди;
- фактический тип фиксируется как `targetmodule`;
- запись переводится в `created`;
- новый activity не создаётся.

Если найден модуль неподдерживаемого типа с тем же `idnumber`, запись получает `failed` с сообщением о конфликте. Автоматическое удаление или замена существующего модуля запрещены.

Перед повторным `/sync` записи со статусом `created` также сверяются с Moodle. Удаление связанного
`assign` или `quiz` переводит элемент и очередь обратно в `pending`; повторное подтверждение создаёт
отсутствующий activity заново. Полностью `synced` очередь при сохранённых activity обрабатывается как
no-op и не вызывает `/sync` повторно.

## 9. Ограничение одного экземпляра

Создать второй `mod_modeussync` нельзя ни автоматически, ни вручную.

Ограничение реализуется несколькими уровнями:

1. `UNIQUE(course)` в таблице `modeussync`.
2. Moodle Lock API с ключом курса перед автоматическим созданием.
3. Проверка существующего экземпляра до вызова Moodle module API.
4. Capability `mod/modeussync:addinstance` не назначается стандартным ролям.
5. Форма добавления отклоняет режим создания нового экземпляра и разрешает только редактирование существующего.
6. `mod_modeussync_add_instance()` принимает вызов только внутри внутреннего request-local контекста системного `instance_manager`.
7. Стандартная попытка администратора добавить activity через chooser завершается контролируемой ошибкой, даже если capability администратора формально позволяет открыть форму.

Удаление системного экземпляра через обычное управление activity также запрещается внутренней проверкой `delete_instance`. Разрешённый административный сценарий удаления должен использовать отдельный внутренний контекст. Если экземпляр отсутствует при наличии актуальной очереди, механизм восстановления создаёт его повторно.

Полное удаление курса распознаётся независимо от web/CLI entry point по стеку core-функций удаления и не блокируется. Событие `course_deleted` удаляет принадлежащие курсу записи очереди, чтобы задача восстановления не обрабатывала осиротевшие `courseid`.

## 10. Интерфейс и права доступа

### 10.1. Доступ

Просматривать страницу могут пользователи с:

```text
mod/modeussync:view
```

Создавать элементы могут пользователи, имеющие одновременно:

```text
mod/modeussync:manage
moodle/course:manageactivities
```

Стандартно эти права назначаются роли преподавателя с правом редактирования и администратору. Студенты не получают доступ.

Ссылка на activity не должна отображаться пользователям без `mod/modeussync:view`. Прямой URL также проверяет login, контекст курса и capability.

### 10.2. Страница activity

Для каждого задания отображаются:

- название;
- максимальная оценка;
- dropdown типа;
- текущее состояние;
- результат или безопасное сообщение об ошибке;
- ссылка на созданный Moodle activity, если он существует.

Dropdown:

- по умолчанию: «Задание»;
- альтернативный вариант: «Тест»;
- недоступен после создания элемента.

Кнопка внизу страницы:

- «Создать» — если есть необработанные элементы;
- «Повторить создание» — если есть ошибки создания;
- «Повторить отправку» — если все элементы созданы, но `/sync` завершился ошибкой;
- неактивна, если очередь уже имеет состояние `synced`.

Кнопка обрабатывает все элементы, которые ещё не имеют состояния `created`. Отдельные checkbox для частичного выбора в первой версии не предусматриваются.

## 11. Создание Moodle activity

Создание реализуется отдельными фабриками:

```text
activity_factory_interface
assign_activity_factory
quiz_activity_factory
```

Общий координатор не формирует универсальный объект для всех модулей. Каждая фабрика отвечает за обязательные и безопасные настройки своего Moodle activity.

### 11.1. Общие свойства

Для обоих типов:

- курс — текущий курс очереди;
- секция — «Задания из Modeus»;
- название — `courseData.name`;
- `course_modules.idnumber` — `courseData.id`;
- максимальная оценка — `courseData.grade`;
- activity создаётся видимым;
- сроки и ограничения не задаются, если их нет в контракте;
- после создания сохраняется `course_modules.id`;
- создание выполняется через штатный Moodle module API.

Валидный входной элемент обязан содержать:

- непустой `id`;
- непустой `name`;
- числовой `grade > 0`.

Некорректный элемент сохраняется для аудита, получает состояние `failed` и блокирует финальный `/sync`.

### 11.2. `mod_assign`

Создаётся задание:

- без дат начала, срока и отсечки;
- без обязательного заявления об отправке;
- без командной отправки;
- без workflow оценивания;
- без включённых способов отправки ответа;
- с максимальной оценкой из `courseData.grade`.

Эта конфигурация сохраняет смысл текущей реализации и позволяет преподавателю позже настроить способы отправки.

Поле `assign.grade` в поддерживаемой версии Moodle хранит только целое значение. Поэтому фабрика
`assign` принимает `courseData.grade` только без дробной части и никогда не округляет его молча.
Для дробного максимума преподаватель должен выбрать `quiz`; попытка создать `assign` остаётся в
состоянии `failed` и может быть повторена после смены типа.

### 11.3. `mod_quiz`

Создаётся пустой тест:

- без вопросов;
- без дат открытия и закрытия;
- без ограничения времени;
- с одной разрешённой попыткой;
- с методом итоговой оценки «Лучшая оценка»;
- с максимальной оценкой из `courseData.grade`;
- с остальными параметрами, явно инициализированными значениями по умолчанию установленной версии `mod_quiz` и зафиксированными интеграционным тестом фабрики.

`quiz.sumgrades` остаётся равным нулю до добавления вопросов. Преподаватель самостоятельно добавляет вопросы и при необходимости меняет настройки теста.

Создание или генерация вопросов из Modeus не входит в область решения.

## 12. Процесс ручного создания

После отправки формы:

1. проверяются `require_login`, контекст course module, capability и `sesskey`;
2. получается Moodle lock по `courseid`;
3. очередь повторно загружается из БД;
4. проверяются допустимые значения `assign` и `quiz`;
5. выбранные значения сохраняются для ещё не созданных элементов;
6. очередь переводится в `processing`;
7. каждый элемент повторно проверяется по `course_modules.idnumber`;
8. отсутствующий элемент создаётся соответствующей фабрикой;
9. успешный элемент получает `created`, `coursemoduleid` и `createdby`;
10. ошибочный элемент получает `failed` и `lasterror`;
11. после обработки пересчитывается состояние очереди;
12. lock освобождается.

Один общий запрос может дать частичный результат. Успешно созданные элементы не откатываются и не удаляются.

## 13. Уведомление SyncService

SyncService не меняется. После успешного создания или обнаружения всех элементов вызывается существующий метод:

```php
\tool_modeussync\service\SyncService::send_sync_courses()
```

Контракт `POST /sync` сохраняется:

```json
[
  {
    "id_modeus": "modeus-course-id",
    "id_lms": "moodle-course-idnumber"
  }
]
```

В `/sync` не передаются:

- Modeus ID отдельных заданий;
- Moodle `cmid`;
- выбранный тип;
- максимальная оценка;
- пользователь, создавший элементы.

Это ограничение относится только к прямому контракту SyncService `POST /sync`.

Созданные `assign` и `quiz` продолжают участвовать в существующей фоновой синхронизации `tool_modeussync\task\push_courses`. Эта задача передаёт сведения о модулях курса в LmsAdapter:

```json
{
  "id": 456,
  "lmsIdNumber": "modeus-assignment-id",
  "name": "Название задания",
  "moduleTypeId": "assign"
}
```

Здесь:

- `id` — Moodle `course_modules.id` (`cmid`);
- `lmsIdNumber` — Modeus ID, сохранённый в `course_modules.idnumber`;
- `name` — название созданного элемента;
- `moduleTypeId` — фактический тип `assign` или `quiz`, выбранный преподавателем.

Таким образом, новый процесс не отменяет передачу `cmid` и типа элемента по существующему каналу `push_courses -> LmsAdapter`. Он только не добавляет эти поля в неизменяемый контракт `POST /sync`.

Сам `mod_modeussync` является техническим интерфейсом, а не учебным элементом РМУП. Поэтому `push_courses` должен исключать его:

- из списка модулей курса, отправляемых в LmsAdapter;
- из списка доступных `moduleTypes`, публикуемых в LmsAdapter.

`id_lms` сохраняет текущую семантику и содержит `course.idnumber`, а не числовой `course.id`.

Вызов `/sync` выполняется только когда все элементы очереди имеют состояние `created`.

## 14. Частичные ошибки и повторные попытки

### 14.1. Ошибка создания отдельного activity

- уже созданные элементы остаются в Moodle;
- успешные записи остаются `created`;
- ошибочная запись становится `failed`;
- очередь возвращается в `pending`;
- `/sync` не вызывается;
- повторная попытка создаёт только отсутствующие элементы.

### 14.2. Ошибка `/sync`

- созданные activity не удаляются;
- элементы остаются `created`;
- очередь становится `sync_failed`;
- сохраняется безопасное сообщение ошибки;
- следующая попытка повторяет только `/sync`.

### 14.3. Прерывание процесса

Персистентное состояние `processing` не считается доказательством активного запроса. При новой попытке после получения lock координатор:

- повторно проверяет Moodle по `idnumber`;
- связывает фактически созданные элементы;
- оставшиеся записи возвращает в `pending` или `failed`;
- продолжает процесс без создания дубликатов.

### 14.4. Ошибка создания `mod_modeussync`

Очередь остаётся сохранённой. Повторное событие или административная задача восстановления снова вызывает идемпотентный `instance_manager`.

## 15. Использование `SyncService` из web-контекста

Текущий `SyncService` использует `mtrace()`, что рассчитано прежде всего на cron и CLI.

Перед использованием из страницы `mod_modeussync` клиент должен получить контекстно-безопасное логирование:

- HTTP-транспорт, настройки, заголовки, таймауты и публичные методы сохраняются;
- вывод не должен попадать в HTML-ответ преподавателю;
- cron может продолжать писать через `mtrace`;
- web-контекст использует Moodle logging/debugging без вывода секретов;
- API key никогда не записывается в лог;
- тело ответа логируется с действующим ограничением длины.

Допустимый вариант — внедряемый logger с реализациями для cron и web. Публичные методы `send_created_courses()` и `send_sync_courses()` сохраняют обратную совместимость.

## 16. Публичные границы компонентов

`tool_modeussync` предоставляет стабильные классы:

- репозиторий очередей;
- сервис идемпотентного сохранения ответа SyncService;
- DTO очереди и элементов;
- событие о появлении заданий;
- SyncService client;
- сервис формирования payload `/sync`.

`mod_modeussync` не должен:

- напрямую разбирать необработанный ответ `/new-course`;
- читать настройки API самостоятельно;
- дублировать HTTP-клиент;
- записывать данные напрямую в таблицы `tool_modeussync`, минуя его публичный сервис;
- изменять существующие контракты SyncService.

## 17. Версии и зависимость

Для согласованного выпуска:

```php
// tool_modeussync/version.php
$plugin->version = 2026071600;
```

```php
// mod_modeussync/version.php
$plugin->version = 2026071600;
$plugin->requires = 2022112800;
$plugin->component = 'mod_modeussync';
$plugin->dependencies = [
    'tool_modeussync' => 2026071600,
];
```

Версия `tool_modeussync` должна быть увеличена до версии, содержащей таблицы очереди, публичные сервисы и событие.

## 18. Миграция существующих данных

Массовая ретроспективная миграция старых курсов не требуется.

При следующем получении `courseData`:

- существующий `mod_assign` с совпадающим `course_modules.idnumber` связывается с элементом очереди;
- элемент отмечается как `created`;
- создаётся единственный `mod_modeussync`, если его ещё нет;
- преподаватель видит уже созданные элементы;
- если SyncService ещё ожидает подтверждение, преподаватель может отправить `/sync` без повторного создания.

Существующие задания не преобразуются в `quiz` и не удаляются.

## 19. Безопасность

Обязательные меры:

- проверка `sesskey` для изменяющих запросов;
- проверка course context и capabilities;
- Moodle Lock API для конкурентных операций;
- валидация `externalid`, названия, оценки и типа;
- параметризованные запросы через Moodle DML;
- экранирование названий и ошибок при выводе;
- отсутствие API key в HTML, событиях и логах;
- отсутствие сырых exception trace в интерфейсе;
- сохранение технических деталей только в серверном журнале;
- запрет произвольного имени Moodle-модуля из POST-данных.

## 20. События и аудит

Следует регистрировать Moodle events:

- задания SyncService сохранены;
- ручное создание начато;
- activity создан;
- создание activity завершилось ошибкой;
- очередь полностью создана;
- `/sync` успешно отправлен;
- `/sync` завершился ошибкой.

События должны содержать идентификаторы курса, очереди и элемента, но не API key и не полный конфиденциальный payload.

## 21. Тестирование

### 21.1. `tool_modeussync`

Unit-тесты:

- разбор валидного `results[*].courseData`;
- отклонение ответа без `results`;
- валидация `id`, `name`, `grade`;
- идемпотентный upsert;
- сохранение выбранного `targetmodule`;
- повторное открытие `synced` при новом задании;
- отсутствие повторного открытия при идентичном ответе;
- формирование неизменного payload `/sync`.

Интеграционные тесты:

- сохранение нескольких курсов из одного batch;
- повторный ответ без дубликатов;
- обработка существующего Moodle activity по `idnumber`;
- публикация события после сохранения.
- передача созданных `assign` и `quiz` через штатный поток `push_courses`;
- исключение технического `mod_modeussync` из `push_courses` и публикуемого списка `moduleTypes`.

### 21.2. `mod_modeussync`

Unit-тесты:

- mapping dropdown в `assign` и `quiz`;
- запрет иных значений;
- переходы состояний;
- повтор `/sync` без повторного создания;
- request-local guard создания экземпляра.

Интеграционные тесты:

- автоматическое создание одного экземпляра;
- повторное событие не создаёт второй экземпляр;
- конкурентное создание защищено lock и unique index;
- ручная попытка добавить экземпляр отклоняется;
- создание `assign`;
- создание пустого `quiz`;
- корректный `course_modules.idnumber`;
- частичная ошибка и повторная попытка;
- конфликт существующего unsupported-модуля;
- запрет доступа студенту;
- успешный вызов `/sync` только после создания всех элементов.

## 22. Критерии приёмки

Решение считается реализованным, если:

1. `pull_courses` больше не создаёт задания автоматически.
2. `courseData` сохраняется в Moodle до закрытия cron-задачи.
3. При наличии заданий на курсе появляется ровно один `mod_modeussync`.
4. Второй экземпляр нельзя создать cron-задачей, конкурентным запросом или вручную.
5. Повторный ответ `/new-course` не создаёт дубликаты очереди или activity.
6. Преподаватель видит задания в столбик и dropdown для каждого необработанного элемента.
7. Dropdown содержит только `assign` и `quiz`, по умолчанию выбран `assign`.
8. Кнопка «Создать» создаёт все ещё не созданные элементы.
9. `assign` получает название, Modeus ID и целочисленную максимальную оценку из `courseData`; дробное значение не округляется и блокирует создание `assign`.
10. `quiz` создаётся без вопросов и получает название, Modeus ID и максимальную оценку из `courseData`.
11. Частичная ошибка не удаляет успешно созданные элементы.
12. Повторная попытка не создаёт дубликаты.
13. `/sync` вызывается только после создания всех элементов.
14. Payload `/sync` полностью соответствует существующему контракту.
15. При ошибке `/sync` элементы сохраняются, а отправку можно повторить отдельно.
16. `mod_modeussync` использует настройки и публичный SyncService-клиент `tool_modeussync`.
17. Студент не может открыть страницу или запустить создание.
18. API key не выводится в интерфейс и журналы.
19. Созданные `assign` и `quiz` передаются в LmsAdapter штатной задачей `push_courses` вместе с `cmid`, `idnumber`, названием и фактическим типом.
20. Технический `mod_modeussync` не передаётся в LmsAdapter как модуль курса и не публикуется как доступный `moduleType`.

## 23. Вне области решения

В первую версию не входят:

- изменение SyncService;
- новый контракт `/new-course` или `/sync`;
- изменение контракта `POST /sync` для добавления `cmid` и выбранного типа элемента;
- создание вопросов в `quiz`;
- поддержка Moodle-модулей кроме `assign` и `quiz`;
- преобразование уже созданного `assign` в `quiz`;
- автоматическое удаление заданий, исчезнувших из нового ответа;
- использование `ukd_grade_helper`;
- массовая миграция всех исторических курсов;
- изменение механизма синхронизации оценок `push_grades`.

## 24. Последствия решения

Положительные последствия:

- преподаватель контролирует тип создаваемого элемента;
- данные SyncService не теряются между cron и ручным действием;
- повторные ответы и повторные попытки безопасны;
- SyncService и его контракты не меняются;
- интеграционные настройки остаются в одном компоненте;
- архитектура расширяется новыми фабриками без изменения очереди.

Издержки:

- требуется изменение обоих плагинов;
- добавляются таблицы и состояния очереди;
- необходимо поддерживать восстановление после частичных ошибок;
- создание курса и окончательное подтверждение SyncService становятся асинхронными по отношению к действиям преподавателя;
- потребуется адаптировать логирование `SyncService` для web-контекста.
